# Linear-Sequence CRDT

For representing indexable sequences, that can be part of the stored object tree, we're going to use a non-interleaving
variant of the Linear Sequence (LSeq) CRDT algorithm.

In the past we already researched such an algorithm - it was described on [this blog post](https://www.bartoszsypytkowski.com/non-interleaving-lseq/).

## Key points

**LSeq** algorithm works by producing a **variable-length byte string** and assigning it to a value at a corresponding index.
The byte strings are generated in such a way that an entry at position `N` has a byte string that's lexically lower than
byte string of entry at position `N+1` and lexically higher than the byte string of entry at position `N-1`. Knowing the 
byte string sequences of previous and next indices, we can generate such bytestrings.

Generated bytes are always higher than `0` (with is used as a delimiter in object path) and lower than `255`.

### Non-interleaving

We achieve non-interleaving variant by producing a sequences of so called `Segment`s. Segments are encoded as a pair of
`(PID, u32)`, where `PID` is a peer identifier of a peer who appended this segment to a LSeq sequence, while `u32` is
an incrementally monotonic positive (>0) integer used for ordering.

Without compression each segment would take at least 8 bytes. For that reason both PID and sequence number are prefix 
length varint encoded.
